What is Cloud Tiering?
Cloud tiering is a storage optimization strategy that classifies data based on usage patterns 
and moves it between different storage tiers to balance performance and cost.
Frequently accessed data, often called hot data, remains on high-performance storage,
for fast access, while infrequently accessed or archival data, known as cold data, 
is moved to cheaper, slower cloud storage. This approach helps organizations reduce ,
storage costs without compromising access to critical data.

Here are real, commonly used AzCopy commands that come up a lot in interviews and practical work.

🔹 1️⃣ Upload files (Local → Azure Blob)
azcopy copy "C:\data" "https://<storage-account>.blob.core.windows.net/<container>?<SAS-token>" --recursive

👉 What it does:

Uploads all files from local folder to Azure Blob Storage
--recursive → includes subfolders
🔹 2️⃣ Download files (Azure → Local)
azcopy copy "https://<storage-account>.blob.core.windows.net/<container>?<SAS-token>" "C:\data" --recursive

👉 Downloads entire container to your local machine

🔹 3️⃣ Copy between containers (within Azure)
azcopy copy "https://srcaccount.blob.core.windows.net/container1?<SAS>" \
             "https://destaccount.blob.core.windows.net/container2?<SAS>" \
             --recursive

👉 Useful for:

Migration between storage accounts
Backup scenarios
🔹 4️⃣ Sync command (very important ⭐)
azcopy sync "C:\data" "https://<storage-account>.blob.core.windows.net/<container>?<SAS>" --recursive

👉 What makes this special:

Only copies changes (delta sync)
Faster than copy for repeated operations
🔹 5️⃣ Copy single file
azcopy copy "C:\data\file.txt" "https://<storage-account>.blob.core.windows.net/<container>?<SAS>"

👉 Simple one-file upload

🔹 6️⃣ Using login instead of SAS (more secure)
azcopy login
azcopy copy "C:\data" "https://<storage-account>.blob.core.windows.net/<container>" --recursive

👉 Uses Azure AD authentication instead of SAS token

🔹 7️⃣ Copy from URL (cross-cloud scenario)
azcopy copy "https://source-url/file.csv" \
             "https://<storage-account>.blob.core.windows.net/<container>?<SAS>"

👉 Example:

AWS S3 → Azure
Public dataset → Azure

types of storages in azure

Azure storage isn’t just one thing—it’s a set of services designed for different kinds of 
data and access patterns. Here are the main types you should know (these are the ones expected 
in interviews):

☁️ Types of Storage in Azure
1️⃣ Azure Blob Storage
👉 Best for: Unstructured data
Stores: images, videos, backups, logs, documents
Highly scalable and cost-effective
Access via REST APIs or URLs

🧠 Example:
Store app uploads, backups, media files

2️⃣ Azure File Storage
👉 Best for: File shares (like a network drive)
Uses SMB/NFS protocols
Can be mounted like a normal drive
Works with on-prem via Azure File Sync

🧠 Example:
Shared folders for teams or applications

3️⃣ Azure Queue Storage
👉 Best for: Messaging between applications
Stores messages/tasks
Helps build asynchronous systems

🧠 Example:
App sends a message → background service processes it

4️⃣ Azure Table Storage
👉 Best for: Structured NoSQL data
Key-value store (schema-less)
Fast and scalable

🧠 Example:
User profiles, metadata, logs

5️⃣ Azure Disk Storage
👉 Best for: Virtual machines
Provides disks for Azure VMs
Types: HDD, SSD, Premium SSD

🧠 Example:
OS disk and data disk for virtual machines

🧠 Easy way to remember
Storage Type	Use
Blob	Files (unstructured data)
File	Shared folders
Queue	Messaging
Table	NoSQL data
Disk	VM storage
🎯 Interview tip

Instead of just listing, say:

👉
“Azure provides multiple storage services: Blob for unstructured data, File for shared access 
via SMB, 
Queue for messaging, Table for NoSQL data, and Disk for virtual machine storage.”
